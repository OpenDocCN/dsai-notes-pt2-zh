# 04：智能体编码模式 🚀

在本节课中，我们将探讨软件开发领域正在经历的戏剧性转折点。我们将学习开发者如何从传统的代码编写者，转变为管理多个高度自主AI智能体的“智能体管理者”。核心在于掌握一项新的元技能——**上下文工程**。

## 从代码编写者到智能体管理者

回顾软件开发的历史，我们经历了从个人英雄式开发到大型专业团队的演变。2023年左右，我们进入了AI辅助编码时代。如今，我们正迈入一个更为激进的阶段。

开发者的核心角色已彻底转变。重点不再是单个人类编写代码，或与Copilot等工具逐行协作。关键在于单个开发者管理多个高度自主的AI智能体的输出。你正在从**编码者**转变为**智能体管理者**。

我们本次探讨的资料来源，聚焦于当前实现这一转变的工具，特别是名为**Claude Code**的命令行工具。它灵活且不受特定范式约束，但要将其转化为高生产力平台，需要一个系统化的方法。这就是我们今天的任务：帮助你这位未来的智能体管理者，理解在此取得成功所需的新元技能——**上下文工程**。

掌握此技能的开发者，其客观和主观生产力相比传统开发方式提升了高达**2.5倍**。来自Stock App等AI原生公司的核心洞见是：成功的衡量标准已经改变。好的代码现在只是**良好上下文**的副产品。

## 理解生产力飞跃：范式转变

2.5倍的提升看似难以置信，但数据似乎支持这一点。例如，Stock App报告其开发者每周人均成功合并约**10.6个**拉取请求，而行业平均水平接近**1个**。主观上，他们报告的生产力提升比手动编码高约2.5倍，并且比那些仅仅将AI工具嫁接到旧流程上的公司高出约两倍。

要理解为何可能实现这种飞跃，必须理解这个范式转变。我们过去一直专注于**提示工程**——编写那个完美的指令。但当自主智能体尝试执行多步骤计划时，这种方法完全失效。

新的核心挑战变成了**上下文工程**。它关乎定义系统的整个知识库、设定行为约束、建立防护栏，以便智能体能够自主运行而不会造成损害。资料强调，应将你的代码仓库（尤其是单体仓库）视为人类和智能体共享的工作空间。这意味着，设计文档、项目计划、架构指南等自然语言产物，突然变得与代码本身同等重要。你正在为智能体储备“食物”。

## 渐进式分层工作流

这种转变如何实际改变开发生命周期？它创建了一个**渐进式分层工作流**，一个自上而下、持续构建上下文的工作流。

工作流始于人类作为架构师。人类定义高级业务需求和目标。接着，人类进行委派。第一次委派通常是给一个设计智能体，由其根据需求起草详细的设计文档。

那么，之后人类的角色是什么？是直到代码完成都袖手旁观吗？绝对不是。人类必须审查该设计文档的正确性和可行性。只有在人类批准后，智能体才能进入下一步：将设计转化为一系列任务，即实施计划。这个审查关口至关重要，因为修复一个有缺陷的设计，远比在后期修复有缺陷的代码要便宜得多。

计划获批后，智能体开始实施并编写代码。但资料强调了一个关键步骤，他们称之为**安全网**。这在智能体开发中是非协商的。智能体倾向于局部解决问题，但由于其上下文视野有限，可能会破坏不相关的系统。因此，必须创建、提交广泛的、会失败的测试。这些测试作为关键的客观约束。如果智能体无法通过现有的测试套件，它就不能提交代码。

然后，循环以审查和更新结束。智能体管理者审核功能，同时确保智能体更新所有相关文档：动态的设计文档、README文件、架构图等。这样，下一个处理该项目的智能体将继承一个完美、准确的上下文。这是一个持续自我完善的系统，正是这种持续流动解锁了2.5倍的增益。

## 核心机制：Claude.md 文件

既然上下文是“货币”，我们如何在像Claude Code这样的工具中实际创建和管理它呢？核心机制是一个名为 **`claude.md`** 的特殊文件。

你应该将此文件视为针对特定代码仓库的、为智能体**个性化定制的指令手册**。Claude Code会在任何会话开始时自动将此文件拉入上下文，因此智能体的“第一个念头”就由这些约束塑造。所以，这是放置项目特定规则的理想位置——不仅是构建什么，还有如何构建。

例如，你可以记录核心工具函数、构建过程、如何精确运行测试套件、常见的bash命令、特定的仓库规范，或任何你希望Claude了解的非标准行为。它非常灵活，你可以将此文件放在根目录、父目录或子目录中，这对于管理大型单体仓库中的上下文至关重要。你甚至可以使用 `claude init` 命令快速生成一个起始文件。

如果这个文件是不断完善的工程产物，团队实际会花多少时间来维护它？当它与代码不同步时会发生什么？这正是智能体管理者的主要挑战。它必须被视为一等公民，就像你的“基础设施即代码”一样。团队通常会自动化维护工作，或指定特定的智能体定期审计文档。当它过时时，智能体会依赖过时的假设，从而导致工具使用错误或代码被拒绝。

为了确保遵守，团队通常会将 `claude.md` 文件通过“提示改进器”处理，或者更简单地在关键指令前添加强调词，如 **`IMPORTANT:`** 或 **`You must...`**，以确保智能体给予这些指令最大权重。

## 控制与安全

这个工具直接在终端中运行，可以写入文件、执行bash命令。这有点吓人。确实如此，而这正是Claude Code固有的保守主义发挥作用的地方。它要求对任何修改系统的操作获得**明确许可**。这是它的看门人功能。

默认情况下，是的。它会在写入文件或执行命令前请求批准。那么，作为智能体管理者，当你想委派一个大型但安全的任务时，如何管理这种摩擦呢？你可以管理权限。使用 `claude permissions` 命令或直接编辑设置文件来预先批准某些操作。例如，允许 `edit` 让智能体自行修改文件；或者你可以通过类似 `bash:git:commit` 的配置实现非常精细的控制，允许智能体提交代码而无需持续监督，但限制其访问敏感的生产工具。这种精细度对安全至关重要。

说到工具，它继承了你的bash环境。但如果你有自定义的CI工具，比如一个专有的部署脚本，智能体不会知道它的存在，除非你明确告知它，或要求它对那个工具运行 `help` 命令。智能体需要你本地基础设施的文档。你还可以使用**模型上下文协议**（MCP）来扩展其能力，以集成外部系统，如专门的测试框架。

## 高效工作流模式

现在，让我们进入实际的日常工作。以下是定义这个新时代的成功工作流。

首先是**探索、计划、编码、提交**工作流，尤其适用于非平凡问题。这个工作流的核心是战略性地管理计算资源。你告诉智能体读取所有文件，但明确指示它**先不要开始编码**。然后你要求它制定一个计划。这可以防止智能体仓促地采用一些肤浅的、次优的解决方案。

这时，那些触发短语就派上用场了。是的，比如 **`think`**、**`think hard`** 甚至 **`ultra-think`**。这些短语向模型发出信号，要求其为规划和评估分配逐渐更高的计算预算。你实际上是在用计算资源换取更彻底的内部思考。这是在开始编写代码前，强制系统分析影响的好方法，否则你可能不得不丢弃已写的代码。

另一个强大的模式是**测试驱动的智能体开发**。这非常有效。Claude在有一个明确的客观约束需要解决时表现最佳。指示这个非意识系统“先失败”很有趣，但这是秘诀吗？给予它一个明确的目标来求解？是的，目标是确定性。工作流要求首先指示Claude编写全面的测试，你确认这些测试失败，然后提交这些测试。接着，你指示Claude编写代码，目的仅仅是让那些已提交的测试通过。它不断迭代，直到测试套件全部通过。这给了智能体管理者对输出结果的完全信心。

对于前端或视觉工作，有**代码-截图-结果-迭代**工作流。你给Claude视觉目标，比如将设计稿图像拖放到终端中。然后，使用像Puppeteer MCP这样的工具，Claude可以对其实时实现进行截图，并迭代修改代码，直到视觉结果与设计稿匹配。这种迭代优化对质量至关重要，通常只需两到三个周期，输出质量就会显著提高。它模仿了人类设计师打磨作品的方式，但以机器速度进行。

## 专业化与多智能体工作流

这把我们带到了真正革命性收益发生的地方：**专业化与多智能体工作流**。

为什么单个智能体在拥有所有上下文的情况下表现不佳？单个智能体难以有效地为不同角色分离上下文。一个试图同时扮演编码者、审查者和测试者的单个智能体会感到困惑。因此，你需要一个验证循环。这就是为什么验证循环如此关键。你运行两个并行实例：一个Claude编写代码，第二个Claude（甚至是通过MCP服务器连接的不同LLM，如Gemini）专门负责审查或测试那个输出。职责分离。

我们有现实世界的证据。Stock App发现，当他们运行这个循环时，第二个模型（本例中是Gemini）在预先发现安全问题和细微错误方面，明显优于编写代码的Claude实例。

如果你在同一个仓库上有真正独立的任务，开发者会使用像Git worktrees这样的工具。简单来说，它允许智能体同时检出代码的隔离工作空间，而不会干扰主分支。这实现了真正的并行执行。

手动管理所有这些的复杂性一定很大。确实令人望而却步，这就是为什么我们看到这些**元编程框架**迅速崛起，它们自动化了编排过程。例如，SuperC提供了一个配置层，包含数十个命令和专门的智能体，如用于任务管理的PM智能体、深度研究智能体。或者Awesome套件，它建立了一个完整的、编排好的AI开发团队：技术主管协调者、代码考古学家等。这些框架自动检测仓库并配置到你的Claude工具中的最佳映射，抽象掉了管理复杂性，让智能体管理者可以专注于高级目标。

## 跨组织的影响

然而，最引人注目的是这如何在整个组织中流动，而不仅仅是软件团队。正是在这里，顿悟时刻不断发生。

让我们看看数据基础设施团队的经验。这是一个复杂的系统任务：Kubernetes集群开始故障，这是一场危机。他们没有时间筛选日志或等待网络专家。相反，他们将仪表盘和错误日志的截图直接输入Claude Code。视觉上下文，结合智能体分析系统配置的能力，使其能够即时诊断问题——一个特定Pod上微妙的IP地址耗尽问题。更重要的是，它提供了确切的bash命令来修复问题，完全绕过了对专业人类的需求。这非常了不起。

在光谱的另一端，是增长营销团队——一个传统的非技术团队。那里的一名团队成员使用两个子智能体（一个负责标题，一个负责描述）构建了一个智能体工作流。他们在几分钟内生成了数百个不同的谷歌广告变体，创意输出增加了10倍，从而实现了指数级更快的A/B测试和迭代。

法律团队的故事同样引人注目。一名非开发者在**一小时内**为一位有语言障碍的家人构建了一个功能性的、预测性文本辅助应用程序。这验证了非开发者通过成为有效的智能体管理者，也能取得非常有意义的成果。

我们也看到它在高度技术性的领域，如安全工程，他们将事件解决时间从10-15分钟减少到大约5分钟，仅仅通过指示Claude通过堆栈跟踪追踪控制流。对于产品设计师，实现大型状态变更或将设计稿转化为原型，现在速度快了两到三倍。

## 给未来智能体管理者的启示

那么，这一切对于你——现在的智能体管理者——意味着什么？关键似乎是**迭代和具体化**。你需要将Claude视为一个迭代伙伴，并尽早、经常地进行过程纠正。使用具体的指令（而不仅仅是模糊的意图）可以显著提高成功率。你必须积极管理这个过程。

你提到了过程纠正工具。是的，你总是强制要求在编码前制定计划。但如果智能体在工具调用或文件编辑过程中开始失控，你可以按**Esc键**立即中断它并重新引导其计划。你拥有直接控制权。你甚至可以**双击Esc**跳回历史记录并编辑你刚刚提供的提示。这种程度的管理控制定义了你的成功。

最终的结论是：开发者的核心技能已从编码语法转变为**管理复杂性**。工作是定义上下文、管理智能体并审计其工作。我们已经看到AI如何指数级提高生产力。但考虑到智能体错误的巨大威力和潜在影响范围，你所拥有的最深层的专业知识——你的领域知识、业务规则——现在被需要用来定义规则集和审计工作。

因此，你，智能体管理者，需要思考六个月后。你正在为一个能力将呈指数级增长的模型构建系统。你今天需要建立什么样的上下文、什么样的明确安全措施、什么样的具体 `claude.md` 文件和健壮的TDD模式，以确保当管理的智能体变得更快、更自主时，质量依然可控？这正是需要你最深层人类专业知识的工作。
